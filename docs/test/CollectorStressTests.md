# Collector stress tests

Дата подготовки отчета: 2026-05-25.

Документ описывает стресс-тесты для `HSMDataCollector`, которые проверяют поведение коллектора под высокой параллельной нагрузкой и при проблемах на стороне HSM-сервера: медленные ответы, HTTP 500, обрывы соединения и последующее восстановление.

## Где находится тест

Основной файл тестов:

`src/collector/HSMDataCollector.Tests/CollectorStressTests.cs`

Тесты написаны для проекта:

`src/collector/HSMDataCollector.Tests/HSMDataCollector.Tests.csproj`

Тестовый проект таргетит `.NET Framework 4.8`, а проверяемая библиотека собирается под `net472` и `net6.0`.

## Что эмулирует тестовый сервер

В тестах используется локальный `HttpListener`, который поднимается на свободном порту `127.0.0.1`.

Коллектор отправляет в него реальные HTTP-запросы по тем же путям, которые использует production-клиент:

- `/api/sensors/commands`
- `/api/sensors/addOrUpdate`
- `/api/sensors/list`
- `/api/sensors/double`
- другие endpoint-ы сенсорных значений, если они будут задействованы тестом

Сервер считает:

- общее количество запросов;
- количество command-запросов;
- количество data-запросов;
- количество ответов `500 Internal Server Error`;
- количество принудительно оборванных соединений;
- количество медленных ответов;
- суммарный размер полученных request body;
- максимальное число одновременных запросов.

## Негативные сценарии

Фейковый сервер умеет намеренно ломать часть запросов:

| Сценарий | Как эмулируется | Что проверяется |
| --- | --- | --- |
| HTTP 500 | Каждый N-й запрос получает `500` | Коллектор не падает и продолжает работать после серверных ошибок |
| Обрыв соединения | Сервер вызывает `Response.Abort()` | Коллектор переживает разрыв сокета и не зависает навсегда |
| Сервер долго не отвечает | Сервер задерживает ответ на заданный интервал | `HttpClient.Timeout` и остановка коллектора работают предсказуемо |
| Восстановление сервера | После плохих ответов сервер снова отвечает `200 OK` | Коллектор продолжает доставлять последующие данные |

Первые несколько запросов всегда проходят успешно. Это нужно, чтобы сенсоры успели зарегистрироваться, а дальше тест уже начинает портить соединение.

## Почему добавлен RequestTimeout

Для сценария "сервер не отвечает" нужен верхний предел ожидания HTTP-запроса. Иначе зависший сервер может держать `HttpClient` до стандартного таймаута .NET, который слишком длинный для тестов и неудобен в production.

Добавлена опция:

```csharp
public TimeSpan RequestTimeout { get; set; } = TimeSpan.FromSeconds(30);
```

В стресс-тестах она уменьшается до `2-3` секунд, чтобы быстро проверять зависшие запросы.

Также исправлено поведение адреса сервера: если `ServerAddress` явно содержит `http://`, клиент больше не переписывает схему в `https://`. Это нужно для локального тестового `HttpListener`. Если схема не указана, поведение остается прежним: используется HTTPS.

## Быстрый стресс-тест

Тест:

`Collector_survives_transient_server_failures_under_parallel_load`

Запускается всегда вместе с обычным `dotnet test`.

Параметры нагрузки:

| Параметр | Значение |
| --- | --- |
| Количество double-сенсоров | 48 |
| Количество worker-задач | 24 |
| Значений на worker | 800 |
| Всего значений | 19 200 |
| Период сборки пакетов | 100 ms |
| Размер пакета | 250 значений |
| Максимальный размер очереди | 30 000 |
| HTTP timeout | 2 секунды |

Настройки фейкового сервера:

| Параметр | Значение |
| --- | --- |
| Успешных запросов в начале | 3 |
| HTTP 500 | каждый 7-й запрос |
| Обрыв соединения | каждый 17-й запрос |
| Медленный ответ | каждый 11-й запрос |
| Задержка медленного ответа | 200 ms |

Проверки:

- сервер получил хотя бы один запрос;
- были command-запросы на регистрацию сенсоров;
- были data-запросы со значениями;
- реально произошли `500` ответы;
- реально произошли обрывы соединения;
- реально произошли медленные ответы;
- `Dispose()` коллектора завершился за 20 секунд.

Команда запуска:

```powershell
dotnet test .\src\collector\HSMDataCollector.Tests\HSMDataCollector.Tests.csproj --no-restore --filter "FullyQualifiedName~CollectorStressTests.Collector_survives_transient_server_failures_under_parallel_load"
```

Локальный результат:

```text
Passed! - Failed: 0, Passed: 1, Skipped: 0, Total: 1, Duration: 3 s
```

## Длинный 10-минутный стресс-тест

Тест:

`Collector_runs_for_ten_minutes_against_flaky_server_under_sustained_load`

По умолчанию он пропущен, чтобы обычный CI и локальные быстрые проверки не занимали 10 минут.

Для запуска нужно явно включить переменную окружения:

```powershell
$env:HSM_COLLECTOR_RUN_LONG_STRESS='1'
dotnet test .\src\collector\HSMDataCollector.Tests\HSMDataCollector.Tests.csproj --no-restore --filter "FullyQualifiedName~CollectorStressTests.Collector_runs_for_ten_minutes_against_flaky_server_under_sustained_load"
```

Параметры нагрузки:

| Параметр | Значение |
| --- | --- |
| Длительность по умолчанию | 10 минут |
| Количество double-сенсоров | 96 |
| Количество worker-задач | `max(8, Environment.ProcessorCount * 2)` |
| Размер внутренней пачки worker-а | 500 значений |
| Период сборки пакетов | 100 ms |
| Размер пакета | 250 значений |
| Максимальный размер очереди | 100 000 |
| HTTP timeout | 3 секунды |

Настройки фейкового сервера:

| Параметр | Значение |
| --- | --- |
| Успешных запросов в начале | 3 |
| HTTP 500 | каждый 19-й запрос |
| Обрыв соединения | каждый 43-й запрос |
| Медленный ответ | каждый 29-й запрос |
| Задержка медленного ответа | 1 секунда |

Проверки:

- сервер получил больше 100 запросов;
- сервер получил больше 50 data-запросов;
- реально произошли `500` ответы;
- реально произошли обрывы соединения;
- `Dispose()` коллектора завершился за 30 секунд;
- managed memory growth меньше 256 MB после полного GC.

## Короткая проверка механизма длинного теста

Чтобы проверить, что длинный тест не только находится в `Skip`, но и реально работает, он был запущен в укороченном режиме через `HSM_COLLECTOR_STRESS_MINUTES`.

Команда:

```powershell
$env:HSM_COLLECTOR_RUN_LONG_STRESS='1'
$env:HSM_COLLECTOR_STRESS_MINUTES='0.05'
dotnet test .\src\collector\HSMDataCollector.Tests\HSMDataCollector.Tests.csproj --no-restore --filter "FullyQualifiedName~CollectorStressTests.Collector_runs_for_ten_minutes_against_flaky_server_under_sustained_load"
Remove-Item Env:\HSM_COLLECTOR_RUN_LONG_STRESS
Remove-Item Env:\HSM_COLLECTOR_STRESS_MINUTES
```

`0.05` минуты - это примерно 3 секунды активной генерации нагрузки. Полный тест использует 10 минут, если `HSM_COLLECTOR_STRESS_MINUTES` не задан.

Локальный результат укороченного запуска:

```text
Passed! - Failed: 0, Passed: 1, Skipped: 0, Total: 1, Duration: 8 s
```

## Проверка сборки библиотеки

Библиотека была отдельно собрана под оба поддерживаемых таргета.

Команды:

```powershell
dotnet build .\src\collector\HSMDataCollector\HSMDataCollector.csproj -f net472 --no-restore
dotnet build .\src\collector\HSMDataCollector\HSMDataCollector.csproj -f net6.0 --no-restore
```

Результат:

```text
net472: Build succeeded, 0 errors.
net6.0: Build succeeded, 0 errors.
```

Для `net6.0` остались существующие warnings по Windows-only API (`EventLog`, `PerformanceCounter`, `Registry`, `ServiceController`). Эти предупреждения не появились из-за стресс-тестов.

## Что считается успешным прохождением

Стресс-тест считается успешным, если:

- коллектор не падает при параллельной отправке большого количества значений;
- обрывы соединения не приводят к зависанию очередей;
- медленные ответы не блокируют остановку процесса навсегда;
- HTTP 500 не ломают последующие отправки;
- `Dispose()` завершается в ограниченное время;
- при длительном прогоне managed memory не растет без ограничений;
- тестовый сервер реально фиксирует плохие сценарии, а не просто отвечает `200 OK`.

## Что этот тест не гарантирует

Тест не проверяет абсолютную доставку каждого значения до сервера. При намеренных `500`, обрывах соединения и ограниченном размере очереди часть значений может быть потеряна или отброшена по design-логике очереди.

Тест не заменяет production soak-test на реальном HSM-сервере. Он проверяет устойчивость клиента и очередей в контролируемых отказах локального сервера.

Полный 10-минутный прогон не был запущен в текущей сессии, чтобы не держать рабочий цикл лишние 10 минут. Механизм длинного теста был проверен укороченным запуском, а сам 10-минутный режим готов к запуску через переменные окружения.
