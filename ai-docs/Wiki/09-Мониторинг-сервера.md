# Мониторинг сервера HSM

HSM мониторит себя, используя ту же библиотеку HSMDataCollector, которую предоставляет пользователям. Все данные самомониторинга собираются в выделенный продукт под названием "HSM Server Monitoring", который создаётся автоматически при первом запуске.

> Не используйте этот продукт для хранения данных вашего приложения — он зарезервирован для внутренних нужд сервера.

---

## Структура дерева датчиков

```text
HSM Server Monitoring
├── computer/               ← Стандартные системные датчики (ЦП, ОЗУ, Диск и т.д.)
│   └── module/             ← Стандартные датчики процесса
├── Database/               ← Размер базы данных и статистика
├── Clients/                ← Трафик API и частота обновлений датчиков по клиентам
├── Telegram Bot/           ← Статистика уведомлений Telegram
└── ...                     ← Другие внутренние узлы
```

---

## Системные датчики и датчики процессов (COMPUTER / MODULE)

Стандартные датчики, собираемые HSMDataCollector, так же как любое пользовательское приложение получало бы от `AddAllDefaultSensors()`.

**На Windows:**
* Общая загрузка ЦП
* Свободная ОЗУ (МБ)
* .NET GC time
* ЦП процесса, память, количество потоков
* Свободное место на диске + прогноз
* Последнее обновление Windows / перезапуск / версия
* Количество ошибок и предупреждений в журнале приложений

**На Linux:**
* Общая загрузка ЦП, свободная ОЗУ
* ЦП процесса, память, количество потоков
* Свободное место на диске + прогноз

## Узел DATABASE

Датчики размера базы данных обновляются раз в день.

| Датчик | Описание |
| :--- | :--- |
| History data size | Размер базы данных истории датчиков (недельные папки) |
| Journals data size | Размер базы данных журнала/аудита |
| Config data size | Сумма баз данных EnvironmentData + ServerLayout + Snapshots |
| Total data size | Общий размер всех баз данных combined |

Все размеры указаны в МБ.

**Что содержит каждая база данных:**
* `EnvironmentData` — метаданные: папки, продукты, датчики, пользователи, ключи доступа
* `ServerLayout` — конфигурация панели мониторинга: графики, панели, макеты
* `Snapshots` — текущее состояние дерева датчиков: время последнего обновления, состояния TTL
* `History` — исторические значения датчиков (разделены на недельные папки)
* `Journals` — записи журнала аудита по датчикам

## Узел CLIENTS

Статистика трафика API по каждому подключённому клиенту (экземпляр DataCollector или вызывающий REST API). Обновляется в реальном времени при поступлении запросов.

Каждый клиент получает собственный подузел под `Clients/`:

| Датчик | Единица | Описание |
| :--- | :--- | :--- |
| Clients requests count | запросов/сек | Общее количество публичных API-запросов от этого клиента |
| Sensors updates | датчиков/сек | Количество полученных значений датчиков |
| Traffic In | КБ/сек | Входящий объём данных от этого клиента |
| Traffic Out | КБ/сек | Исходящий объём ответов этому клиенту |

Также есть узел `Total`, который агрегирует всех клиентов combined.

---

## Узел TELEGRAM BOT

Мониторинг активности уведомлений бота Telegram HSM.

| Датчик | Описание |
| :--- | :--- |
| Total | Частота отправленных уведомлений (в минуту) |
| Errors | Ошибки бота Telegram (неудачные отправки, ошибки API) |
| Messages/<имя-чата> | Последнее отправленное сообщение для каждого подключённого чата Telegram |

## Доступ к данным

Откройте продукт HSM Server Monitoring в веб-интерфейсе, чтобы увидеть все датчики. Вы можете настроить политики оповещений для любого из этих датчиков — например:

* Оповещение, если общий размер данных превышает 10 ГБ
* TTL-оповещение, если размер истории перестаёт обновляться (проблема серверра)
* Оповещение, если Errors в узле Telegram Bot не пуст
