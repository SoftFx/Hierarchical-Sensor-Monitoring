# Расписания оповещений

Расписание оповещений определяет рабочие часы для доставки уведомлений. Если к политике привязано расписание, оповещения отправляются только в настроенные временные окна — в остальное время уведомления тихо подавляются.

Расписания можно использовать повторно: одно расписание может быть назначено нескольким политикам оповещений для разных датчиков.

---

## Управление расписаниями

Перейдите в `Alerts → Schedules` в веб-интерфейсе HSM, чтобы создавать, редактировать или удалять расписания.

Каждое расписание имеет:
* **Name** — отображается в настройках политики
* **Timezone** — все временные окна оцениваются в этом часовом поясе
* **YAML configuration** — тело расписания (формат см. ниже)

---

## YAML-формат

Расписание настраивается с помощью YAML. Редактор проверяет YAML в реальном времени и отображает ошибки.

### Минимальный пример — только будни
```yaml
daySchedules:
  - days: [Mon, Tue, Wed, Thu, Fri]
    windows:
      - start: "09:00"
        end: "18:00"
```

### Полный пример со всеми опциями
```yaml
daySchedules:
  - days: [Mon, Tue, Wed, Thu, Fri]
    windows:
      - start: "09:00"
        end: "13:00"
      - start: "14:00"
        end: "18:00"
  - days: [Sat]
    windows:
      - start: "10:00"
        end: "14:00"

disabledDates:
  - "2025-12-25"
  - "2026-01-01"

overrides:
  enabledDates:
    - "2025-03-08"
  customScheduleDates:
    - date: "2026-04-01"
      windows:
        - start: "10:00"
          end: "12:00"
```

## Справочник полей

### `daySchedules` (обязательно)
Список расписаний по дням недели. Каждая запись покрывает один или несколько дней с их временными окнами.

```yaml
daySchedules:
  - days: [Mon, Wed, Fri]   # дни, к которым применяется эта запись
    windows:
      - start: "09:00"      # формат: HH:mm (24-часовой)
        end: "17:00"
```

Названия дней:
| Значение | День |
|----------|------|
| Mon      | Понедельник |
| Tue      | Вторник |
| Wed      | Среда |
| Thu      | Четверг |
| Fri      | Пятница |
| Sat      | Суббота |
| Sun      | Воскресенье |

Правила:
* Требуется как минимум одна запись с хотя бы одним днём и одним окном
* Формат времени: `HH:mm` (24-часовой, например, `09:00`, `23:30`)
* `start` должен быть раньше `end`
* Минимальная длительность окна: 1 минута
* Окна внутри одной записи дня не должны пересекаться
* Допускается несколько окон в день (например, для обеденного перерыва)

### `disabledDates` (необязательно)
Список дат, в которые оповещения никогда не отправляются, даже если они выпадают на обычный рабочий день.

```yaml
disabledDates:
  - "2025-12-25"
  - "2026-01-01"
  - "2026-05-09"
```
Формат даты: `yyyy-MM-dd`

### `overrides` (необязательно)
Исключения из обычного расписания.

#### `enabledDates`
Даты, которые считаются обычными рабочими днями, даже если они не входят в `daySchedules` (например, суббота, отрабатываемая вместо праздника).
```yaml
overrides:
  enabledDates:
    - "2025-11-08"   # Суббота отработана вместо выходного
```
Используются временные окна из соответствующей записи `daySchedules` для этого дня недели.

#### `customScheduleDates`
Даты с полностью пользовательским временным окном, переопределяющим как обычные, так и правила отключения.
```yaml
overrides:
  customScheduleDates:
    - date: "2026-04-01"
      windows:
        - start: "10:00"
          end: "14:00"
```
Каждая `customScheduleDate` должна иметь либо `windows`, либо `scheduleType` (но не оба одновременно).

> Дата не может одновременно находиться в `enabledDates` и `customScheduleDates`.

---

## Приоритет принятия решений

При проверке, должно ли сработать оповещение в заданное время, HSM оценивает условия в следующем порядке:
1. `customScheduleDates` — если у даты есть пользовательское переопределение, используются эти окна (или `scheduleType`)
2. `enabledDates` — если дата принудительно включена, используются обычные окна `daySchedules` для этого дня недели
3. `disabledDates` — если дата отключена, оповещение подавляется
4. `daySchedules` — используется обычное расписание для этого дня недели

Если ни одно из условий не соответствует текущему дню, оповещение подавляется.

---

## Часовой пояс

Поле `Timezone` устанавливается в интерфейсе HSM (не в YAML-теле). Используются стандартные идентификаторы часовых поясов IANA:
* `Europe/Minsk`
* `Europe/Moscow`
* `UTC`
* `America/New_York`

Все временные окна в YAML интерпретируются в этом часовом поясе. HSM хранит и сравнивает время в UTC внутренне и преобразует его в локальное время для оценки расписания.

---

## Правила валидации

YAML-редактор применяет эти правила перед сохранением:
* `daySchedules` должен присутствовать и не быть пустым
* Каждое расписание дня должно содержать хотя бы один день и одно окно
* Формат времени должен быть `HH:mm`
* `start < end`, минимальное окно — 1 минута
* Отсутствуют перекрывающиеся окна в одной записи дня
* Отсутствуют дубликаты дат в `customScheduleDates`
* Дата не может быть одновременно в `enabledDates` и `customScheduleDates`
* Даты должны быть в диапазоне 2000–2100

---

## Назначение расписания политике

В редакторе политики оповещений нажмите **Schedule** и выберите существующее расписание оповещений из списка. Политика будет срабатывать только в настроенные рабочие часы.

Это отдельный параметр от настройки **Repeat** (которая контролирует частоту повторных уведомлений, пока условие активно). Оба параметра могут работать одновременно.
