# Основные понятия

На этой странице объясняются фундаментальные строительные блоки HSM.

---

## Датчик

Датчик — это базовая единица мониторинга. Он имеет путь, тип и хранит поток значений во времени.

### Путь

Путь идентифицирует датчик внутри продукта. Это строка, разделённая косыми чертами:

`ServiceName/component/metric`

**Примеры:**
- `MyService/database/query_time_ms`
- `MyService/api/requests_per_second`
- `worker-1/process/memory_mb`

**Правила:**
- Строка после последнего `/` — это имя датчика
- Всё до него формирует иерархию папок
- Пути уникальны в пределах продукта
- Новый датчик создаётся автоматически при первой отправке данных в новый путь
- Максимальная глубина пути: 10 уровней (настраивается администратором)

### Ключ

Каждый запрос к API должен включать ключ доступа. Ключ определяет, к какому продукту относятся входящие данные и какие права есть у вызывающей стороны.

Ключи создаются для каждого продукта в веб-интерфейсе: Продукты → Ключи доступа.

---

## Продукт

Продукт — это контейнер верхнего уровня для датчиков. Он представляет одно приложение, сервис или систему, за которой ведётся наблюдение.

Каждый продукт имеет:
- Один или несколько ключей доступа для приёма данных
- Дерево датчиков всех путей, по которым отправлялись данные
- Чаты оповещений по умолчанию — чаты Telegram, которые получают оповещения по умолчанию

---

## Папка

Папки объединяют продукты в группы. Они используются для:
- Организации продуктов по команде, окружению или домену
- Привязки чатов Telegram — чат, подключённый к папке, получает оповещения от всех продуктов в этой папке
- Применения разрешений ключей доступа на более высоком уровне

---

## Статус датчика

Каждый датчик имеет вычисленный статус на основе своих политик оповещений:

| Статус  | Значок | Значение                                      |
|---------|------|----------------------------------------------|
| Ok      | 🟢   | Активных оповещений нет                               |
| Error   | 🔴   | Сработало хотя бы одно условие политики оповещения  |
| OffTime | 🕑   | Датчик не отправлял данные в течение окна TTL |

**Приоритет статусов (от низшего к высшему):** OffTime → Ok → Error

Родительский узел показывает статус с наивысшим приоритетом среди всех нижележащих датчиков.

---

## Состояние датчика

Отдельно от статуса датчик имеет состояние, контролирующее, активен ли он:

| Состояние     | Значение                                           |
|-----------|---------------------------------------------------|
| Available (Доступен) | Нормальная работа                                  |
| Muted (Приглушён)     | Датчик активен, но все оповещения подавляются    |
| Blocked (Заблокирован)   | Датчик permanently отключен                    |

---

## Типы датчиков

HSM поддерживает 11 типов датчиков:

| Тип        | Значение                  | Сценарий использования                                      |
|-------------|------------------------|---------------------------------------------------------------|
| Bool        | true / false           | Сервис запущен, флаг функции, бинарное состояние     |
| Int         | целое число                | Глубина очереди, количество потоков, количество ошибок        |
| Double      | число с плавающей запятой         | ЦП %, память МБ, время отклика               |
| String      | текст                   | Последнее сообщение об ошибке, строка статуса             |
| TimeSpan    | длительность               | Длительность запроса, время безотказной работы                      |
| Version     | номер версии         | Версия приложения (например, 2.1.0)              |
| Rate        | double как скорость/скорость   | Запросов в секунду, КБ/сек                   |
| Enum        | целое число, сопоставленное с именами| Конечный автомат, выбор режима                |
| IntegerBar  | агрегированный int         | Время отклика за 5-минутное окно              |
| DoubleBar   | агрегированный double      | Выборки ЦП за 5-минутное окно              |
| File        | содержимое файла           | Отчёты, логи, снимки конфигурации        |

Bar-датчики собирают множество отдельных измерений за настраиваемое временное окно (по умолчанию: 5 минут) и отправляют агрегированную статистику: min, max, mean, count, первое значение, последнее значение.

---

## Комментарий и статус значения

Каждое значение датчика может携带 необязательный комментарий (произвольный текст) и явную переопределённую статус:
- status: `0` — Ok
- status: `1` — Error

Если статус не задан, HSM вычисляет его на основе активных политик оповещений.

---

## Разрешения ключей доступа

Ключи доступа имеют два уровня разрешений:

| Разрешение | Может делать                        |
|------------|-------------------------------|
| Отправка данных  | POST значений датчиков через API    |
| Чтение данных  | GET истории датчиков через API    |

Ключ может иметь одно или оба разрешения. Ключами управляют на уровне продукта пользователи с ролью Manager.

---

## TTL (Time-To-Live)

TTL — это тайм-аут, после которого датчик считается неактивным. Если в течение окна TTL не поступает значение, статус датчика меняется на OffTime и срабатывает оповещение TTL (если оно настроено).

TTL можно установить:
- Для каждого датчика в настройках датчика
- Через `SensorOptions.TTL` в DataCollector
- Через шаблон оповещений, применённый ко многим датчикам сразу

См. [Обзор оповещений](10-Оповещения-Обзор.md) для настройки оповещений TTL.
