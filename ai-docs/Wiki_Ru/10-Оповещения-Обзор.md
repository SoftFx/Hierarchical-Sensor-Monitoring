# Обзор оповещений

Оповещения — ключевая функция HSM. Вы определяете правила (политики), которые описывают, когда и как получать уведомления об изменениях состояния датчика, неактивности или пороговых значениях.

---

## Ключевые понятия

| Понятие | Описание |
|---|---|
| **Политика** | Одно правило оповещения: если [условие] тогда [уведомить] |
| **Условие** | Что запускает оповещение: порог значения, изменение статуса, неактивность |
| **Получатель** | Куда отправить оповещение: чат Telegram |
| **Шаблон** | Текст сообщения уведомления с динамическими переменными |
| **Расписание** | Когда могут срабатывать оповещения: рабочие часы, интервалы повтора |
| **Период подтверждения** | Как долго условие должно сохраняться перед оповещением |
| **TTL** | Оповещение, когда датчик перестаёт отправлять данные |

---

## Как работают политики

Каждый датчик может иметь одну или несколько политик оповещений. При поступлении нового значения датчика HSM оценивает все активные политики:

```
Новое значение датчика
    ↓
Проверка AlertSchedule (рабочие часы) → пропуск, если вне часов
    ↓
Оценка всех условий (AND / OR)
    ↓
Ожидание периода подтверждения (если установлен)
    ↓
Отправка уведомления в чаты назначения
```

Политики оцениваются независимо. Несколько политик могут сработать одновременно.

---

## Условия

Каждая политика имеет одно или несколько условий, объединённых логикой `AND` или `OR`.

### Свойства

Свойство определяет, какая часть значения датчика проверяется:

| Свойство | Применяется к | Описание |
|---|---|---|
| `Value` | Все скалярные датчики | Текущее значение датчика |
| `EmaValue` | Скалярные датчики | Экспоненциальное скользящее среднее значения |
| `Status` | Все датчики | Текущий статус датчика (Ok / Error / OffTime) |
| `Comment` | Все датчики | Комментарий, прикреплённый к значению |
| `Min / Max / Mean` | Bar-датчики | Агрегированная статистика за период |
| `Count` | Bar-датчики | Количество значений в баре |
| `FirstValue / LastValue` | Bar-датчики | Первое/последнее значение в периоде бара |
| `EmaMin / EmaMax / EmaMean` | Bar-датчики | EMA агрегированной статистики |
| `Length` | String-датчики | Длина строки |

### Операции

| Операция | Символ | Применимо к |
|---|---|---|
| Меньше | `<` | Числа, TimeSpan, Version |
| Меньше или равно | `<=` | Числа, TimeSpan, Version |
| Больше | `>` | Числа, TimeSpan, Version |
| Больше или равно | `>=` | Числа, TimeSpan, Version |
| Равно | `=` | Числа, строки, TimeSpan, Version |
| Не равно | `≠` | Числа, строки, TimeSpan, Version |
| Содержит | `—` | Строки |
| Начинается с | `—` | Строки |
| Заканчивается на | `—` | Строки |
| Изменился | `—` | Статус, строки |
| Изменился на Error | `—` | Статус |
| Изменился на Ok | `—` | Статус |
| Является Error | `—` | Статус |
| Является Ok | `—` | Статус |

### Цель

Цель — значение, с которым сравнивается свойство. Это может быть:
* **Константа** — фиксированное значение, которое вы указываете (например, `> 90`)
* **Последнее значение** — предыдущее значение датчика (например, "оповещение, если значение изменилось")

## Шаблоны сообщений

Сообщение оповещения — это текстовый шаблон с переменными, которые заполняются в момент срабатывания оповещения.

### Доступные переменные

| Переменная | Описание |
|---|---|
| `$product` | Имя родительского продукта |
| `$path` | Полный путь датчика |
| `$sensor` | Имя датчика |
| `$status` | Текущий статус (со значком) |
| `$prevStatus` | Предыдущий статус |
| `$value` | Текущее значение датчика |
| `$prevValue` | Предыдущее значение датчика |
| `$time` | Время получения значения |
| `$comment` | Комментарий, прикреплённый к значению |
| `$prevComment` | Предыдущий комментарий |
| `$unit` | Единица измерения датчика |
| `$min / $max / $mean` | Статистика bar-датчика |
| `$count` | Количество значений bar-датчика |
| `$firstValue / $lastValue` | Первое/последнее значения bar-датчика |
| `$emaValue` | EMA текущего значения |
| `$emaMin / $emaMax / $emaMean` | EMA статистики бара |
| `$property` | Имя свойства оповещения |
| `$operation` | Символ операции оповещения |
| `$target` | Цель сравнения |

### Примеры шаблонов

```
[$product] $sensor имеет статус $status: $value $unit

⚠️ $path — загрузка ЦП $value% (было $prevValue%)

[$product] $sensor не восстановился. Статус: $status с $time
```

## Получатели

Каждая политика определяет, куда отправлять уведомления.

**Режимы:**
* **От родителя** — использовать чаты Telegram по умолчанию, настроенные на родительском продукте/папке. Дополнительные конкретные чаты можно добавить поверх.
* **Пользовательский** — отправлять только в явно перечисленные чаты.
* **Все чаты** — отправлять во все чаты Telegram, зарегистрированные в HSM.
* **Пусто** — без уведомления (политика только меняет статус датчика).

См. [Настройка Telegram](14-Настройка-Telegram.md) для подключения чатов Telegram.

## Планирование

### Простой повтор

Управляет частотой повторного срабатывания оповещения, пока условие остаётся активным:

| Режим | Описание |
|---|---|
| Немедленно | Сработать один раз, не повторять |
| Каждые 5 / 10 / 15 / 30 минут | Повтор с интервалом |
| Ежечасно | Повтор каждый час |
| Ежедневно | Повтор раз в день |
| Еженедельно | Повтор раз в неделю |

**Мгновенная отправка** — при включении первое уведомление срабатывает немедленно при срабатывании условия, независимо от интервала повтора.

### Рабочие часы (Расписание оповещений)

Для более сложного подавления можно назначить расписание оповещений политике. Это позволяет ограничить оповещения конкретными днями и временными окнами.

Расписания управляются отдельно в `Alerts → Schedules` и могут использоваться повторно в нескольких политиках.

См. [Расписания оповещений](12-Расписания-оповещений.md) для полной справки.

## Период подтверждения

По умолчанию оповещение срабатывает, как только условие выполнено. С периодом подтверждения условие должно оставаться истинным в течение указанного времени перед отправкой оповещения.

**Сценарий использования:** Датчик ЦП кратко скачет до 100% на 2 секунды. Без подтверждения вы получили бы шумное оповещение. С периодом подтверждения в 1 минуту только устойчивые скачки запускают оповещения.

## Оповещения TTL

Оповещение TTL срабатывает, когда датчик не отправлял данные в течение настроенного интервала.

**Как это работает:**
* Вы устанавливаете TTL на датчике (например, 5 минут)
* Если значение не поступает в течение этого окна, срабатывает политика TTL
* При возобновлении передачи данных автоматически отправляется уведомление о восстановлении `Ok`
* Вы можете настроить повторные уведомления (каждый час, ежедневно и т.д.) для продолжительного молчания
* Сообщение уведомления включает счётчик повторов (например, `#1`, `#2`), показывающий, сколько раз оповещение повторялось

## Наследование политик

Политики можно определять на разных уровнях иерархии и наследовать вниз:

```
Продукт
  └── Папка
        └── Датчик
```

* **Политика TTL** — наследуется от родительского продукта, если не переопределена на уровне датчика
* **Чаты назначения** — датчики наследуют чаты по умолчанию от родительского продукта; отдельные политики могут добавить дополнительные чаты поверх
* **Шаблоны оповещений** — автоматически применяют политики ко всем датчикам, соответствующим шаблону пути и типу датчика (см. [Шаблоны оповещений](13-Шаблоны-оповещений.md))

---

## Статус датчика

Каждый датчик имеет статус, отражающий результат оценки политики:

| Статус | Значок | Значение |
|---|---|---|
| Ok | 🟢 | Политики не сработали или восстановлены после ошибки |
| Error | 🔴 | Активно хотя бы одно условие политики |
| OffTime | 🕑 | TTL истёк — датчик не отправлял данные |

**Приоритет статусов (по возрастанию):** `OffTime → Ok → Error`
