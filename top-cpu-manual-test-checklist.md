# Чек-лист ручного тестирования: Top CPU processes (таски 1175, 1177, 1179, 1180, 1194, 1196)

Фича Windows-only. Тестировать на Windows-хосте с коллектором/агентом, подключённым к HSM-серверу.
Прогнать **дважды** — для managed (C# collector) и native (`hsm-agent.exe --console`) — и сверить.

---

## 0. Подготовка

- [ ] Windows-машина, коллектор/агент подключён к HSM-серверу
- [ ] В веб-UI открыт узел `Top CPU processes`
- [ ] Открыт Task Manager / Process Explorer для сверки значений
- [ ] Под рукой стресс-тул для нагрузки ядра (напр. загрузить один процесс на 100%)
- [ ] Известны параметры: `count` (сколько сенсоров за тик), `minPercent` (порог), `period`

---

## 1. Базовая корректность (1175 / 1179)

- [ ] Под `Top CPU processes` появляются сенсоры `Top CPU processes/<имя_процесса>`
- [ ] За один тик создаётся **не больше `count`** сенсоров
- [ ] Публикуются только процессы с CPU% ≥ порога (`minPercent`)
- [ ] При нагрузке процесс всплывает наверх, значение CPU% совпадает с диспетчером
- [ ] CPU% не превышает 100% (деление на общее число ядер)
- [ ] На сервере >64 ядер (несколько процессорных групп) CPU% не завышен — ветка `GetActiveProcessorCount(AllProcessorGroups)`
- [ ] Сортировка: по убыванию CPU%, тай-брейк по имени по возрастанию
- [ ] Описание обычного сенсора: `Top **N** CPU consumers by % of machine CPU`

---

## 2. Namespace cap (1179 / 1180)

- [ ] Сгенерировать много уникально названных короткоживущих процессов (батч/CI-подобная нагрузка)
- [ ] Число сенсоров под `Top CPU processes` упирается в `max(count*8, 64)` и **перестаёт расти**
- [ ] Уже отслеживаемые имена продолжают обновляться после достижения cap
- [ ] Реестр сенсоров на сервере не растёт бесконечно

---

## 3. Системные / защищённые процессы (1194) — ключевое

- [ ] Найти процесс без доступного `MainModule`: `System`, `Registry`, `vmmemWSL`, `Secure System`
- [ ] Дождаться/добиться его появления в топе
- [ ] В описании сенсора ровно: `**Path:** _(system process - path unavailable)_`
- [ ] НЕ пустая строка и НЕ путь для такого процесса
- [ ] Для обычного процесса в описании: `**Path:** \`C:\...\process.exe\`` (полный путь в бэктиках)
- [ ] **Нет спама исключений**: понаблюдать несколько минут — в логах коллектора нет повторяющихся исключений MainModule / QueryFullProcessImageName каждый тик
- [ ] Пограничный случай: если имя не резолвилось из-за упора в cap — строки `Path` в описании **нет вообще** (не помечается ошибочно как «system process»)

---

## 4. ASCII-заметка (1196)

- [ ] Открыть описание системного сенсора на не-UTF-8 хосте (системная локаль не-1252) или проверить байты описания на сервере
- [ ] Дефис в заметке — обычный ASCII `-`, никаких кракозябр (mojibake)
- [ ] Текст заметки в managed и native **байт-в-байт идентичен**

---

## 5. Жизненный цикл (1194 — правился CollectorQueueShutdownTests)

- [ ] Start → идёт сбор → Stop корректно останавливает фоновый цикл (`RunLoopAsync`) без зависаний
- [ ] Прогнать Init/Start/Stop несколько раз подряд
- [ ] Нет необработанных исключений при остановке
- [ ] Нет утечки задач/потоков после Stop

---

## 6. Сверка managed vs native

- [ ] Имена сенсоров совпадают
- [ ] Описания (включая Path-строку и заметку про system process) совпадают
- [ ] Порядок/выбор топ-N совпадает при одинаковой нагрузке

---

## Порядок прогона (кратко)

1. Запустить коллектор, дать 2–3 периода, проверить появление сенсоров и рост значений под нагрузкой
2. Сверить с Task Manager по 2–3 процессам
3. Проверить системный процесс → заметка про path
4. Стресс уникальными именами → проверить cap
5. Stop/Start несколько раз → посмотреть логи на исключения
6. Повторить на втором коллекторе (native/managed) и сверить
